# AI 协作规则

> 以下规则用于指导 AI 编码助手在本项目中的行为。这些规则会被注入到 AI 上下文中。

## 代码风格

1. 所有新代码必须使用 TypeScript，严格模式
2. 使用 ES Module 语法 (`import`/`export`) 而非 CommonJS
3. 遵循 `5_Dev_Standards/coding_style.md` 中定义的格式化规则
4. 遵循 `5_Dev_Standards/naming_conventions.md` 中定义的命名规范
5. 公共 API 函数/类必须添加 JSDoc 注释

## 提交信息

1. 提交信息必须遵循 Conventional Commits 规范
2. 格式：`<type>(<scope>): <description>`
3. Type 必须是小写（feat, fix, docs, refactor, test, chore）
4. 提交信息使用中文或英文，保持项目内一致

## 文件操作规范

1. **不要删除代码**：除非明确要求，否则不删除已有代码
2. **优先编辑**：修改已有文件用 Edit 工具，不要整文件重写
3. **不创建不必要的文件**：只创建任务明确需要的文件
4. **不修改配置文件**：如 `tsconfig.json`、`package.json` 等，除非任务要求

## 测试要求

1. 新增功能必须包含对应测试
2. 修改逻辑必须更新对应测试
3. 测试覆盖率达到项目目标
4. 运行测试确保通过后再提交

## 安全规则

1. 永远不要在代码中硬编码密钥、Token、密码
2. 使用环境变量管理敏感配置
3. API 输入必须校验（使用 Zod 或其他校验库）
4. SQL 查询必须使用参数化查询，防止注入
5. 所有用户输入在显示前必须转义（防止 XSS）

## 依赖管理

1. 新增依赖前先检查是否已有类似功能的依赖
2. 优先使用项目中已有的库
3. 不要引入体积过大的依赖用于简单功能
4. 更新依赖需要有明确理由

## 架构规则

1. 遵循项目已有的架构模式和分层
2. 新模块应放在正确的位置，不要破坏目录结构
3. 跨模块的变更需要充分考虑影响
4. 不要引入循环依赖

## 文档

1. 重要的设计决策应添加到 ADR
2. API 变更应更新对应的 API 文档
3. 新增的环境变量应更新 README
